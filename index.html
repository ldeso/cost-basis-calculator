Cost Basis Calculator

Compute the cost basis, realized proceeds, and realized P&L of an ERC-20 token holding for any account on Ethereum, Polygon, Base, Optimism, or Arbitrum. Everything runs in your browser — the Alchemy API key is used directly from this page and never sent anywhere else.

Initial state (optional)

Seed a pre-existing balance with a known cost basis, e.g. after a token migration: paste the old token’s remaining amount and remaining cost basis here, then set a start date on or after the migration so pre-migration transfers are excluded.

What is this?

This page computes the cost basis of a single ERC-20 token held by a single wallet on one of five EVM chains. The cost basis is the USD amount you effectively paid to acquire the tokens you still hold; combined with the USD value of tokens you have disposed of, it lets you see your realized profit or loss over the entire lifetime of the position.

Nothing is stored on a server. The page is a static HTML file plus a bundled JavaScript module; the only network calls are made from your browser directly to Alchemy’s public API using the key you paste in.

Inputs

  • Account address — the wallet whose transfers are replayed.
  • Token address — the ERC-20 contract on the selected chain.
  • Chain — Ethereum, Polygon, Base, Optimism, or Arbitrum. The RPC URL is built automatically from the chain and the API key as https://<network>.g.alchemy.com/v2/<ALCHEMY_API_KEY>, so a single Alchemy key that covers the target network is all that is needed.
  • ALCHEMY_API_KEY — used both for the JSON-RPC endpoint and for the Prices API. An Alchemy key is required because the calculator uses Alchemy’s non-standard alchemy_getAssetTransfers method (standard eth_getLogs is generally range-limited on public endpoints and cannot scan entire histories).
  • Method — which lot-selection rule to apply when the account sends tokens out (see below).
  • Initial state (optional) — seed a pre-existing balance with a known cost basis. Intended for use after a token migration: run the calculator on the old token, then feed its last remaining amount and remaining cost basis into these fields for the new token. Combined with a start date that excludes the migration transaction, this carries cost basis across the migration instead of resetting it to migration-day market price.

How it computes the results

  1. Token metadata. The app calls the ERC-20 contract on the selected chain for decimals(), symbol(), and balanceOf(account). The live balance is later compared against the computed remaining amount to flag tokens with non-standard transfer behaviour.
  2. Transfer history. Two paginated alchemy_getAssetTransfers queries — one with toAddress = account and one with fromAddress = account — pull every ERC-20 transfer in and out of the account for the given token, from block 0 to latest. Each transfer is tagged with its direction, amount, block number, transaction hash, and block timestamp.
  3. Ordering. All transfers are merged and sorted strictly by block number, then by Alchemy’s uniqueId as a tie-breaker, so multiple transfers inside the same block stay in a deterministic order.
  4. Daily USD prices. The app calls Alchemy’s Prices API (/prices/v1/<key>/tokens/historical) for the token, on the selected network, with a 1d interval, spanning one day before the first transfer to one day after the last. The time window is chunked into 365-day segments to stay within API limits. Each price point is the token’s USD value at that day.
  5. Pricing each transfer. For each transfer, the nearest daily price point (by absolute time distance) is used as the USD price of that token at that moment.
  6. Replay. Transfers are walked in chronological order. Incoming transfers open a new lot (amount + price + timestamp + tx hash). Outgoing transfers are treated as disposals: their USD value becomes proceeds, and the lot-selection method decides which previously-acquired tokens are considered sold to produce the cost. The difference is the realized P&L for that disposal.
  7. Balance consistency check. After replay, the computed remaining amount is compared against the on-chain balanceOf. A mismatch is surfaced as a warning; it typically indicates rebasing tokens, fee-on-transfer tokens, or mint/burn outside the standard Transfer event.

Lot-selection methods

When an outgoing transfer disposes of N tokens, the method chooses which previously-acquired tokens are considered sold:

  • FIFO (first-in, first-out) — consumes the oldest lots first. Tends to maximize realized gains in rising markets because the lowest-cost tokens are sold first.
  • LIFO (last-in, first-out) — consumes the most recent lots first. Minimizes realized gains in rising markets but is not accepted under all tax regimes.
  • Weighted average — every outgoing transfer is priced at the running average cost per token across all open holdings. There is only ever a single “lot” in this mode, and its price changes each time more tokens come in.

Output

  • Remaining amount — tokens the account still holds after the replay.
  • Remaining cost basis (USD) — total USD attributed to the remaining tokens. Divide by remaining amount to get effective cost per token.
  • Realized proceeds (USD) — sum of USD values of all outgoing transfers at their time-of-transfer prices.
  • Realized cost (USD) — sum of USD costs of the lots consumed by those outgoing transfers, per the selected method.
  • Realized P&L (USD) — proceeds minus cost.
  • Open lots table — every still-held lot (FIFO/LIFO) or the averaged position (weighted average), with links to the acquiring transaction.
  • Realized sales table — every outgoing transfer with its proceeds, matched cost, P&L, and a link to the transaction.

Assumptions & limitations

  • Every outgoing transfer is treated as a disposal at the time-of-transfer USD price. The calculator cannot distinguish a wallet-to-wallet transfer between your own addresses from a true sale — both look identical on-chain.
  • Prices are sampled at daily granularity. Intraday price swings are smoothed.
  • If Alchemy’s price feed does not cover the token on the selected chain, the calculation stops with an error.
  • Tokens with rebases, transfer fees, or other non-standard transfer logic may yield a computed balance that differs from the on-chain balanceOf; the UI flags the mismatch but does not attempt to reconcile it.
  • Gas costs and per-transfer fees are not factored into cost basis or proceeds.
